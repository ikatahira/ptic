Metodologia Ágil — Decomposição na Indústria de Tecnologia

🏃
As grandes empresas de tecnologia (Google, Apple, Meta, startups brasileiras) usam Metodologia Ágil — e ela é baseada em decomposição!

O projeto é dividido em Sprints (ciclos curtos de 1-2 semanas). A cada Sprint, a equipe entrega uma parte funcional do produto.

Vantagens sobre o método tradicional (Cascata):
• Feedback rápido do cliente/usuário
• Mudanças de requisitos são bem-vindas
• Entregas frequentes de valor real
• Problemas descobertos cedo, quando são baratos de corrigir

Frameworks ágeis mais usados: Scrum, Kanban, XP (Extreme Programming)
⚡ +10 XP
✏️ Ciclos curtos de trabalho de 1-2 semanas na metodologia ágil chamam-se:

User Stories — Decomposição do Ponto de Vista do Usuário

📖
Uma User Story (História de Usuário) descreve uma funcionalidade do ponto de vista de quem vai usar o sistema.

Formato padrão da indústria:
"Como [tipo de usuário], eu quero [funcionalidade] para que [benefício]"

Exemplos:
• "Como aluno, eu quero ver meu histórico de XP para que eu acompanhe meu progresso"
• "Como professor, eu quero exportar relatório de notas para que eu entregue na secretaria"
• "Como usuário, eu quero redefinir minha senha para que não fique sem acesso se esquecer"

User Stories focam no valor para o usuário, não na implementação técnica.

Backlog — A Lista Mestra do Projeto

🗂
O Product Backlog é a lista de TUDO que precisa ser feito no produto, organizada por prioridade.

Como funciona:
1. No início do projeto, a equipe decompõe o produto em User Stories
2. Todas as histórias entram no Backlog
3. O Product Owner (dono do produto) ordena por prioridade (1 = mais importante)
4. A cada Sprint, a equipe escolhe as histórias mais prioritárias
5. Histórias concluídas saem, novas histórias podem entrar

Ferramentas para gerenciar Backlog: Trello, Jira, Azure DevOps, GitHub Projects — todas amplamente usadas no mercado de trabalho!
⚡ +10 XP
✏️ A lista priorizada de funcionalidades de um projeto ágil chama-se:
Backlog

MVP — Lançar Rápido, Aprender Rápido

🔄
MVP (Minimum Viable Product / Produto Mínimo Viável) é a versão mais simples do produto que ainda entrega valor real para o usuário.

Filosofia: Lance rápido → Receba feedback real → Itere e melhore.

Exemplos históricos famosos:
• Dropbox: MVP foi um vídeo explicativo. Testou o interesse antes de construir o produto.
• Airbnb: MVP foi um site básico com fotos de 3 quartos no apartamento dos fundadores.
• Facebook: MVP foi só para alunos de Harvard; expandiu gradualmente.

O MVP é a decomposição levada ao extremo: quais são as funcionalidades absolutamente ESSENCIAIS?
⚡ +10 XP
✏️ A versão mais simples de um produto que ainda entrega valor real chama-se:

📋 Planejar um mini-projeto com Backlog e Sprints
Crie o planejamento ágil completo de um app simples usando User Stories, Backlog e Sprints.
⏱ 35 min 📄 Google Docs ⚡ +80 XP
1
Defina o produto: App de lista de tarefas para estudantes do ETEC
2
Escreva 8 User Stories no formato correto (Como... eu quero... para que...)
3
Sugestões: login, criar tarefa, editar tarefa, marcar como concluída, ver histórico, notificação, compartilhar tarefa, exportar PDF
4
Crie o Backlog: ordene as 8 histórias por prioridade (1 = mais urgente/essencial)
5
Sprint 1 (MVP): escolha as 3-4 histórias mais essenciais — o mínimo para o app ser útil
6
Sprint 2 (Melhoria): as próximas 2-3 histórias mais importantes
7
Sprint 3 (Extras): as funcionalidades restantes
8
Salve como Planejamento_Agil_SeuNome e compartilhe com o professor
